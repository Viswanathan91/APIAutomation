## 2. How Execution Starts

```
mvn test -Denv=sit -Dcucumber.filter.tags="@AddPlace"
    │
    ▼
Maven Surefire Plugin
  · passes -Denv=sit to the JVM as a system property
  · discovers TestRunner.java (the only class annotated with @RunWith(Cucumber.class))
    │
    ▼
TestRunner.java
  @CucumberOptions(
    features = "src/test/java/features",   ← scan both .feature files
    glue     = {"stepDefinitions"},         ← scan all step definition classes
    plugin   = "json:target/jsonReports/…"  ← write results as JSON for HTML report
  )
    │
    ▼
Cucumber runtime
  · parses every .feature file
  · filters scenarios by the tag passed on the command line (or runs all if no tag)
  · for each scenario: instantiates step definition classes and runs Given → When → Then → And
    │
    ▼
Results written to target/jsonReports/cucumber-report.json
HTML report generated by maven-cucumber-reporting during mvn verify
```

**Key point:** `TestRunner` has no hardcoded `tags =` value. Tags come entirely from the command line (`-Dcucumber.filter.tags`), so you never need to edit a Java file to change which scenarios run.

## 3. Layer-by-Layer Explanation

### 3.1 Feature Files (What to test)

Feature files are written in **Gherkin** — plain English structured as Given / When / Then. They describe *what* is being tested without any Java code.

```gherkin
Given Add Place Payload with "Viswa house" "English" "No.5 Beach Road"
When  user calls "AddPlaceAPI" with "post" http request
Then  the API call got success with status code 200
And   "status" in response body is "OK"
```

Cucumber reads each line, matches it against registered step definition methods using the expression in the annotation, and calls that method.

### 3.2 Step Definitions (How to test it)

Step definitions are plain Java methods annotated with `@Given`, `@When`, `@Then`, `@And`. The string inside the annotation is a **Cucumber expression** — curly brace tokens like `{string}` and `{int}` are extracted and passed as method arguments.

```java
@Given("Add Place Payload with {string} {string} {string}")
public void add_place_payload_with(String name, String language, String address) {
    res = given().spec(requestSpecification())
                 .body(tb.addPlacePayload(name, language, address));
}
```

There are two step definition classes:

| Class | Handles | Extends |
|---|---|---|
| `StepDefinition` | Places API (rahulshettyacademy.com) | `Utils` |
| `UserStepDefinition` | Post API (jsonplaceholder.typicode.com) | `Utils` |

Both extend `Utils` to inherit `requestSpecification()`, `reqresRequestSpecification()`, and `getJsonPath()`.

### 3.3 Utils (Shared infrastructure)

`Utils` is the base class that provides three things:

**① `requestSpecification()` / `reqresRequestSpecification()`**  
Builds a `RequestSpecification` — a reusable object that pre-configures REST Assured with the base URI, content type, API key, and logging filters. Built once (singleton pattern) and reused for every request in that JVM run.

```
sit.properties ──► getGlobalVariables("baseURI") ──► RequestSpecBuilder ──► RequestSpecification
                    getGlobalVariables("key")
```

**② `getGlobalVariables(key)`**  
Reads a value from the active environment's `.properties` file. The environment is selected by reading the `env` JVM system property (default: `sit`).

**③ `getJsonPath(response, key)`**  
Extracts a value from the JSON response body using dot-notation. Example: `getJsonPath(response, "place_id")` returns the string value of the `place_id` field.

### 3.4 TestDataBuild (Request payload factory)

Constructs request body objects. All hardcoded defaults live here, keeping step definitions free of data construction logic.

```java
// Places API
public AddPlace addPlacePayload(String name, String language, String address) { … }
public String   deletePlacePayload(String placeId) { … }

// Post API
public CreateUserRequest createPostPayload(String title, String body) { … }
```

### 3.5 APIResources (Endpoint registry)

A Java `enum` that stores every API path. Step definitions call `APIResources.valueOf(resource).getResource()` to convert the string from the feature file into the actual path.

```java
AddPlaceAPI("/maps/api/place/add/json")
GetPostAPI ("/posts/{id}")            // {id} is a REST Assured path-param template
```

### 3.6 POJOs (Data models)

Plain Java objects that represent request or response bodies. Jackson (the JSON library) serialises them to JSON for requests and deserialises JSON to them for responses.

| POJO | Direction | Used for |
|---|---|---|
| `AddPlace` + `Location` | Request → JSON | POST /maps/api/place/add/json |
| `CreateUserRequest` | Request → JSON | POST /posts, PUT /posts/{id} |
| `GetUserResponse` | JSON → Response | GET /posts/{id} |

### 3.7 Hooks (Cross-scenario setup)

`Hooks.java` contains a `@Before("@DeletePlace")` method. Cucumber runs it automatically before any scenario tagged `@DeletePlace`.

The `@DeletePlace` scenario needs a `place_id` to delete, but that `place_id` was created by a prior `@AddPlace` run. Hooks solves this by checking if `place_id` is already set and, if not, calling the AddPlace steps directly to create one.

## 4. End-to-End Data Flow — Places API

### 4.1 `@AddPlace` scenario

```
Feature file
  "Add Place Payload with 'Viswa house' 'English' 'No.5 Beach Road'"
        │
        │ Cucumber matches step text → calls method
        ▼
StepDefinition.add_place_payload_with("Viswa house", "English", "No.5 Beach Road")
        │
        │ delegates payload construction
        ▼
TestDataBuild.addPlacePayload("Viswa house", "English", "No.5 Beach Road")
        │ builds AddPlace POJO with all fields populated
        ▼
AddPlace {
  name: "Viswa house", language: "English", address: "No.5 Beach Road",
  accuracy: 50, phone_number: "(+91) 983 893 3937", website: "http://google.com",
  types: ["shoe park", "shop"],
  location: { lat: -38.383494, lng: 33.427362 }
}
        │
        │ REST Assured serialises POJO → JSON via Jackson
        ▼
HTTP POST https://rahulshettyacademy.com/maps/api/place/add/json?key=qaclick123
Content-Type: application/json
Body: { "accuracy":50, "name":"Viswa house", "language":"English", ... }
        │
        │ API returns response
        ▼
Response { "status":"OK", "scope":"APP", "reference":"...", "id":"...", "place_id":"abc123" }
        │
        │ @Then assertion
        ▼
assertEquals(200, response.getStatusCode())           ← status code check
assertEquals("OK",  getJsonPath(response, "status"))  ← JsonPath field checks
assertEquals("APP", getJsonPath(response, "scope"))
        │
        │ @And verify step — cross-validation using the place_id
        ▼
placeId = getJsonPath(response, "place_id")   // stores "abc123" in static field
res = given().spec(requestSpecification()).queryParam("place_id", "abc123")
HTTP GET https://rahulshettyacademy.com/maps/api/place/get/json?key=qaclick123&place_id=abc123
Response { "name":"Viswa house", … }
assertEquals("Viswa house", getJsonPath(response, "name"))
```

### 4.2 `@DeletePlace` scenario — with Hooks

```
Cucumber sees @DeletePlace tag
        │
        ▼
Hooks.beforeScenario() runs FIRST (before any Given/When/Then)
        │
        │ checks: is StepDefinition.placeId null?
        ├── YES → calls m.add_place_payload_with(…) and m.user_calls_with_http_request(…)
        │          to create a place and populate placeId from the AddPlace response
        └── NO  → skips (place already exists from a previous scenario in this run)
        │
        ▼
Feature file: "DeletePlace payload"
        │
        ▼
StepDefinition.delete_place_payload()
        │ uses the static placeId set either by @AddPlace or by the Hook
        ▼
TestDataBuild.deletePlacePayload("abc123")
        │ returns raw JSON string: {"place_id":"abc123"}
        ▼
HTTP POST https://rahulshettyacademy.com/maps/api/place/delete/json?key=qaclick123
Body: {"place_id":"abc123"}
        │
        ▼
Response { "status":"OK" }
assertEquals(200, response.getStatusCode())
assertEquals("OK", getJsonPath(response, "status"))
```

**Why `placeId` is `static`:** Instance fields are reset for every scenario (Cucumber creates a new step definition object per scenario). Making `placeId` static means its value survives across scenarios within the same JVM run, allowing `@DeletePlace` to use the ID created by `@AddPlace`.

## 5. End-to-End Data Flow — Post Management API

### 5.1 `@CreatePost` — all fields present

```
Feature file:
  "Create Post payload with title 'First Post' and body 'Content for first post'"
        │
        ▼
UserStepDefinition.create_post_payload_with_title_and_body("First Post", "Content for first post")
        │
        ▼
TestDataBuild.createPostPayload("First Post", "Content for first post")
        │ returns new CreateUserRequest("First Post", "Content for first post")
        │ Both fields are non-null → @JsonInclude(NON_NULL) keeps both
        ▼
Jackson serialises CreateUserRequest → {"title":"First Post","body":"Content for first post"}
        │
        ▼
HTTP POST https://jsonplaceholder.typicode.com/posts
Body: {"title":"First Post","body":"Content for first post"}
        │
        ▼
Response 201: {"title":"First Post","body":"Content for first post","id":101}
        │
        ▼
assertEquals(201, response.getStatusCode())
assertEquals("First Post",              getJsonPath(response, "title"))   ← mandatory check
assertEquals("Content for first post",  getJsonPath(response, "body"))    ← mandatory check
[Optional check] "id" is present → logs "101"                             ← optional check
```

### 5.2 `@CreatePostOptional` — optional field excluded from request

```
Feature file:
  "Create Post payload with only title 'Title Only'"
        │
        ▼
UserStepDefinition.create_post_payload_with_only_title("Title Only")
        │
        ▼
TestDataBuild.createPostPayload("Title Only", null)   ← body is null
        │
        ▼
Jackson serialises CreateUserRequest:
  · title = "Title Only"  → written to JSON
  · body  = null          → @JsonInclude(NON_NULL) SKIPS this field entirely
        │
        ▼
HTTP POST https://jsonplaceholder.typicode.com/posts
Body: {"title":"Title Only"}          ← "body" key does not appear at all

        │
        ▼
Response 201: {"title":"Title Only","id":101}   ← server echoes back only what was sent
        │
        ▼
assertEquals(201,          response.getStatusCode())
assertEquals("Title Only", getJsonPath(response, "title"))
[Optional check] "body" is present?
  → body.contains("\"body\"") is false
  → prints "[Optional] 'body' is NOT present — skipping validation"
  → test continues without failing
```

### 5.3 `@GetPost` — POJO deserialisation

```
Feature file:
  "I request post with id 1"
        │
        ▼
UserStepDefinition.i_request_post_with_id(1)
  res = given().spec(reqresRequestSpecification())
               .pathParam("id", 1)       ← binds {id} template variable
        │
        ▼
  "user calls 'GetPostAPI' with 'GET' http method"
        │
        ▼
APIResources.GetPostAPI.getResource() → "/posts/{id}"
REST Assured resolves {id} → "/posts/1"

HTTP GET https://jsonplaceholder.typicode.com/posts/1
        │
        ▼
Response 200:
{
  "userId": 1,      ← JSON key
  "id": 1,
  "title": "sunt aut facere…",
  "body":  "quia et suscipit…"
}
        │
        │ "response is deserialized and 'id' is '1'"
        ▼
First call to this step in the scenario:
  deserializedPost = response.as(GetUserResponse.class)
                     │

        │

prints "[Optional] 'title' is present → sunt aut facere…"

```

## 6. The Five Patterns Demonstrated

### Pattern 1 — POJO Serialisation (request body)

**File:** `AddPlace.java`, `TestDataBuild.addPlacePayload()`

REST Assured serialises a Java POJO to a JSON request body automatically when `ContentType.JSON` is set on the `RequestSpecification`. You only write Java setters — no manual JSON string building.

```java
AddPlace ap = new AddPlace();
ap.setName("Viswa house");
// REST Assured calls Jackson internally:
given().spec(requestSpecification()).body(ap)  →  {"name":"Viswa house", …}
```

### Pattern 2 — Optional Request Fields (`@JsonInclude`)

**File:** `CreateUserRequest.java`

```java
@JsonInclude(JsonInclude.Include.NON_NULL)   // class-level annotation
public class CreateUserRequest {
    private String title;
    private String body;   // optional
}
```

When `body` is `null`, Jackson's `NON_NULL` rule omits the field from the JSON output entirely. The caller does not need `if` statements — passing `null` is the convention.

```java
createPostPayload("Hello", null)  →  {"title":"Hello"}          // body key absent
createPostPayload("Hello", "Hi")  →  {"title":"Hello","body":"Hi"}
```

### Pattern 3 — JsonPath Response Validation

**File:** `Utils.getJsonPath()`, `StepDefinition.in_response_body_is()`

JsonPath lets you navigate a JSON response using dot-notation without deserialising into a POJO. Useful for quick field checks.

```java
getJsonPath(response, "status")          // → "OK"
getJsonPath(response, "place_id")        // → "abc123"
getJsonPath(response, "location.lat")    // → "-38.383494"  (nested field)
```

### Pattern 4 — Optional Response Validation

**File:** `UserStepDefinition.optionally_is_present_in_response()`

Some API fields only appear under certain conditions (timestamps, environment-only fields, fields under development). The optional step checks for presence and logs the value, but never causes a test failure.

```java
if (body.contains("\"" + key + "\"")) {
    System.out.println("[Optional] '" + key + "' is present → " + getJsonPath(response, key));
} else {
    System.out.println("[Optional] '" + key + "' is NOT present — skipping validation");
}
```

Contrast with mandatory validation:
```java
assertEquals(expectedValue, getJsonPath(response, key));  // fails if absent or wrong value
```

### Pattern 5 — POJO Deserialisation (`response.as()`)

**File:** `GetUserResponse.java`, `UserStepDefinition.response_is_deserialized_and_field_is()`

`response.as(GetUserResponse.class)` tells REST Assured to deserialise the full JSON response body into a typed Java object. After this call you have compile-time type safety, IDE auto-complete, and no raw string keys.

```java
GetUserResponse post = response.as(GetUserResponse.class);
post.getId()       // int — no casting needed
post.getAuthorId() // int — mapped from "userId" in JSON via @JsonProperty
post.getTitle()    // String
```

`@JsonProperty("userId")` on `authorId` tells Jackson to map the JSON key `"userId"` to the Java field `authorId`. Without it, Jackson would look for a JSON key named `"authorId"` and find nothing.

```java
@JsonProperty("userId")
private int authorId;   // JSON: "userId":1  →  Java: authorId = 1
```

## 7. Environment Switching

The environment controls which base URIs and API keys are used for the entire test run.

```
mvn test -Denv=uat
          │
          ▼
Surefire passes -Denv=uat to the JVM
          │
          ▼
Utils.getGlobalVariables("baseURI")
  String env = System.getProperty("env", "sit");   // → "uat"
  InputStream is = getResourceAsStream("uat.properties");
  p.load(is);
  return p.getProperty("baseURI");                  // → value from uat.properties
          │
          ▼
RequestSpecBuilder.setBaseUri("https://rahulshettyacademy.com")   ← from uat.properties
```

All three properties files have the same keys. In a real project, each file has different values pointing to different hosts:

```
sit.properties   → baseURI=https://sit.internal.company.com
uat.properties   → baseURI=https://uat.internal.company.com
prod.properties  → baseURI=https://api.company.com
```

The `RequestSpecification` is built once per JVM run (singleton pattern), so the environment cannot change mid-run — which is the intended behaviour.

## 8. How to Add a New Endpoint

Follow these four steps — no other files need to change.

**Step 1 — Register the path in `APIResources.java`**
```java
GetOrderAPI("/api/orders/{orderId}"),
```

**Step 2 — Add a request POJO in `src/main/java/pojo/` (if POST/PUT)**
```java
@JsonInclude(JsonInclude.Include.NON_NULL)
public class CreateOrderRequest {
    private String productId;   // required
    private Integer quantity;   // optional — pass null to omit
    …
}
```

**Step 3 — Add a payload builder in `TestDataBuild.java`**
```java
public CreateOrderRequest createOrderPayload(String productId, Integer quantity) {
    return new CreateOrderRequest(productId, quantity);
}
```

**Step 4 — Write a Gherkin scenario in a `.feature` file**
```gherkin
@GetOrder
Scenario: Fetch order by id
  Given I request order with id 42
  When user calls "GetOrderAPI" with "GET" http method
  Then the API call is successful with status code 200
  And response field "productId" is "ABC"
```

Add the matching `@Given` and step methods in the appropriate step definition class.  
If the endpoint belongs to a new service (different base URI), add a new properties key and a new `requestSpecification` method in `Utils`.
